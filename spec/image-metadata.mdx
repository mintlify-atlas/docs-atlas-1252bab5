---
title: "Image metadata labels for pre-built dev containers"
description: "Store dev container configuration directly in image labels using the devcontainer.metadata label, enabling self-contained pre-built development images."
---

When you pre-build a dev container image and push it to a registry, the image and its configuration can become separated. To solve this, the Dev Container Specification lets you embed configuration metadata directly inside the image as a Docker label named `devcontainer.metadata`. Any supporting tool that pulls the image reads this label and merges its contents with the local `devcontainer.json` at the time the container is created. This makes pre-built images self-contained — developers get the right tooling configuration even when no `devcontainer.json` is present in the repository.

## The devcontainer.metadata label

The label value is a JSON string representing an **array** of metadata objects. Using an array means that each image layer — whether from the base image or from a Dev Container Feature installed on top — can append its own entry without needing to re-parse and re-merge the existing label.

```json
[
  {
    "id": "ghcr.io/devcontainers/features/node:1",
    "entrypoint": "/usr/local/share/nvm/nvm-init.sh",
    "onCreateCommand": "echo 'Node feature initialized'"
  },
  {
    "postCreateCommand": "npm install",
    "remoteUser": "vscode",
    "customizations": {
      "vscode": {
        "extensions": ["esbenp.prettier-vscode"]
      }
    }
  }
]
```

For tooling simplicity, a single top-level object (instead of an array with one entry) is also accepted:

```json
{
  "remoteUser": "vscode",
  "postCreateCommand": "npm install"
}
```

<Tip>
  Always prefer the array form in Dockerfiles so that subsequent image builds can append entries without rewriting the label.
</Tip>

## Adding the label in a Dockerfile

Use the `LABEL` instruction in your Dockerfile to embed the metadata. Wrap the JSON value in single quotes and use `\` for line continuation when the value is long.

```dockerfile
FROM mcr.microsoft.com/devcontainers/base:ubuntu

# Install project dependencies
RUN apt-get update && apt-get install -y curl git

LABEL devcontainer.metadata='[{\
  "remoteUser": "vscode",\
  "postCreateCommand": "npm install",\
  "forwardPorts": [3000],\
  "customizations": {\
    "vscode": {\
      "extensions": ["esbenp.prettier-vscode", "dbaeumer.vscode-eslint"]\
    }\
  }\
}]'
```

<Note>
  Dockerfile labels have a line length limit of approximately 65k characters and a total Dockerfile size limit of around 1.3 MB. Using one line per Feature entry keeps you well within these limits.
</Note>

## Merge logic

When a container is created, the tool merges the image label metadata with the contents of any local `devcontainer.json`. The `devcontainer.json` is always considered **last** in the merge, so its values take precedence when ordering matters. Variable substitution in string values happens at the time each value is applied.

| Property | Merge strategy |
|---|---|
| `id` | Not merged. |
| `init` | `true` if any source sets it to `true`. |
| `privileged` | `true` if any source sets it to `true`. |
| `capAdd` | Union of all arrays, duplicates removed. |
| `securityOpt` | Union of all arrays, duplicates removed. |
| `entrypoint` | All values collected into a list. |
| `mounts` | All values collected. For conflicting sources, last wins. |
| `onCreateCommand` | All values collected into a list. |
| `updateContentCommand` | All values collected into a list. |
| `postCreateCommand` | All values collected into a list. |
| `postStartCommand` | All values collected into a list. |
| `postAttachCommand` | All values collected into a list. |
| `waitFor` | Last value wins. |
| `customizations` | Merging delegated to each tool. |
| `containerUser` | Last value wins. |
| `remoteUser` | Last value wins. |
| `userEnvProbe` | Last value wins. |
| `remoteEnv` | Per variable, last value wins. |
| `containerEnv` | Per variable, last value wins. |
| `overrideCommand` | Last value wins. |
| `portsAttributes` | Per port, last value wins. |
| `otherPortsAttributes` | Last value wins. |
| `forwardPorts` | Union of all, duplicates removed. On mapping conflicts, last wins. |
| `shutdownAction` | Last value wins. |
| `updateRemoteUserUID` | Last value wins. |
| `hostRequirements` | Maximum value wins for each field (`cpus`, `memory`, `storage`, `gpu`). |

## How it relates to prebuilt images

The typical prebuilding workflow is:

<Steps>
  <Step title="Author a Dockerfile">
    Write your Dockerfile and embed `devcontainer.metadata` as a `LABEL` with all the configuration your image needs.
  </Step>
  <Step title="Build and push the image">
    Build the image and push it to your registry. The metadata travels with the image layers.
  </Step>
  <Step title="Reference the image">
    In your repository's `devcontainer.json`, reference the pre-built image with the `image` property. You can omit any properties already encoded in the label, or override them as needed.
  </Step>
  <Step title="Open the dev container">
    When a supporting tool creates the container, it reads the image label, merges it with the local `devcontainer.json`, and applies the combined configuration.
  </Step>
</Steps>

```json
{
  "name": "My project",
  "image": "ghcr.io/myorg/myimage:latest"
}
```

Even with a minimal `devcontainer.json` like this, the container gets all of the configuration embedded in the image label — lifecycle scripts, port forwarding, extensions, and more.
